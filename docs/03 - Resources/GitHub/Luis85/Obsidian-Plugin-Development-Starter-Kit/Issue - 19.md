# Proof of Concept
#scoping

## Problem

- I want to have a standard structure for plugin development to switch seamlessly between projects
- I don't want to mess around with Setup and Configuration of the Development Environment
- I want to have a good baseline to quickly start Plugin Development 
- I need guidance to stay on track


## Insight

- I have multiple Plugin Ideas all over the place
- I want to quickly test and validate new ideas and solutions
- I need a place to design, document, develop, test, and publish my solutions
- I want to reduce the amount of tools I use for my day to day work
- I want to use Obsidian as my single point of truth
- I am drowning in Projects and Administrative Tasks instead of creating something valuable
- I want my work build around Issues I want to solve


## Opportunity

By having a simple and easy to use Project Starter-Kit I can cut manual and administrative tasks and focus on constantly delivering a quality product and service trough incrementell improvements and data-driven decisions making cheap change possible.

## Solution - Key Features

### Prototype Platform

As an Obsidian Plugin Developer, I can easily test new Plugin Ideas without the burden of overcomplex processes while maintaining flexibility for continuous improvements by using a solid, well documented boilerplate. 

By starting with the problem, defined by Issues in GitHub to designing a solution scoped with a GitHub Pull Request and importing them into Obsidian, I can bring E-2-E traceability to the whole development lifecycle.


### Scope & Deliverables

- Repo Structure
- Documentation Structure
- Development Workflow
- Development Environment
- Plugin Boilerplate Template
- Stock taking
- Solution Concept

## Acceptance Criteria

- I can repeaditly create a new Plugin Development Project based on a Fork of the Starter-Kit
- I have a Baseline Development Environment ready to start Plugin Development 
- Basic Project Management Deliverables are documented in the Repo
- Basic Product Management Deliverables are documented in the Repo

## Assumptions

- GitHub Pull Requests could be used for Product Increment Definition and Product Management
- GitHub Issues could be used for Problemspace Management
- The Repository Readme could be used as Main Entry Point to the Products Features
- GitHub and Obsidian are the perfect fit for effective Open Source Product Development
- The GitHub Startpage is the first Touchpoint with the Product
- Pull Requests can be used as central Tool for Product Development Iteration Contracts and Stakeholder Communication


## Decisions

- TypeScript for type safety development
- Vite for building and deployment
- Vitest as testing framework
- npm as package manager
- Obsidian > 1.9 to use the Bases Feature
- Git for versioning
- VS Code for development
- GitHub as Git Remote Provider
- GitHub for Issue Management
- GitHub Pull Requests Workflow
